# Работа над задачей

Выполни задачу целиком: контекст, план, реализация, отработка ревью, завершение.

## Когда использовать

- Пользователь просит реализовать `US-XXX` или `ISS-XXX`.
- Пользователь просит взять следующую задачу из backlog и выполнить её.

## Когда не использовать

- Пользователь хочет только обсудить идею или подготовить план без реализации.
- Пользователь просит только ревью, commit или обновление документации.

## Процесс

### 1. Контекст

1. Прочитай `project_state.md`.
2. Прочитай целевой story/issue:
   - `docs/stories/US-XXX.md` или `docs/stories/done/US-XXX.md`;
   - `docs/issues/ISS-XXX.md` или `docs/issues/done/ISS-XXX.md`.
3. Если задача не указана явно, предложи верхнюю задачу из backlog.
4. При старте задачи обнови `project_state.md`: перенеси задачу в `Current Sprint`.
5. Найди релевантные файлы кода и документации до начала изменений.

### 2. Планирование

1. Исследуй код и определи точки изменения.
2. Если есть неустранимая неоднозначность, задай пользователю 1-2 коротких вопроса.
3. Подготовь самодостаточный план в `docs/plans/{ID}-plan.md`, используя формат из скилла `write-plan`.
4. Остановись и дождись явного одобрения плана перед реализацией.

### 3. Реализация

1. Работай по vertical slices из утверждённого плана.
2. Пиши или обновляй тесты вместе с кодом.
3. Добавляй логирование и обработку ошибок там, где меняется поведение системы.
4. После каждого логического блока запускай релевантные проверки.
5. Не выходи за рамки задачи без отдельного согласования.

### 4. Отработка ревью

1. Если есть файл ревью данной задачи в `docs/reviews/`, прочитай его целиком.
2. По каждому finding реши: принять и исправить или отклонить с обоснованием.
3. Помни, что ты отвечаешь за результат, поэтому не принимай замечания бездумно.

### 5. Завершение

1. После явной команды пользователя на commit финализируй трекинг и собери один осмысленный commit.
2. Для `US-XXX`:
   - перенеси story в `docs/stories/done/`;
   - обнови в story секции `Technical Notes` и `Related Files`, если они изменились по итогам реализации.
3. Для `ISS-XXX`:
   - перенеси issue в `docs/issues/done/`.
4. Всегда:
   - обнови `project_state.md`;
   - перенеси задачу из `Current Sprint` в `Done`, если она велась там;
   - дополни `Грабли` или архитектурные заметки, если в ходе работы появились устойчивые проектные выводы.
5. Очистка:
   - удали связанный файл плана из `docs/plans/`;
   - удали связанные review-файлы в `docs/reviews/`, если они привязаны к ID задачи.
6. Если в ходе задачи появились устойчивые знания о внешнем API, лимитах или нестандартном поведении:
   - зафиксируй детали в `docs/notes/`;
   - в `project_state.md` оставь только короткий trigger и, при необходимости, ссылку на note.

## Ограничения

- Не начинай реализацию до одобрения плана.
- Не коммить без явного разрешения пользователя.
- Все изменения по задаче: код, тесты и трекинг, собирай в один commit.
- Не превращай `project_state.md` в длинную документацию.
- Не добавляй абстракции без реальной необходимости.
