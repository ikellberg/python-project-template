# Аудит проекта

Оцени текущее состояние проекта как независимый аудитор. Главный результат:
отчёт о здоровье проекта с конкретными findings и рекомендациями.

## Когда использовать

- Пользователь просит аудит кода, архитектуры, тестов или документации.
- Пользователь хочет оценить качество или здоровье проекта.
- Периодическая проверка между задачами.
- Пользователь спрашивает "что с качеством", "где проблемы", "что улучшить".

## Когда не использовать

- Пользователь просит проверить конкретные изменения, diff или commit — это `code-review`.
- Пользователь просит написать или исправить код — это `implement-task`.

## Процесс

1. **Определи scope.** Спроси или определи из контекста: полный аудит или конкретная область (код, архитектура, тесты, документация). Если пользователь не уточнил — проведи полный.
2. **Изучи контекст.** Прочитай `project_state.md`, архитектурный документ, структуру проекта.
3. **Пройди чеклист** выбранной области систематически.
4. **Контекстный анализ.** Для каждого потенциального finding подтверди контекстом: действительно ли это проблема в данном проекте, или pattern match ложный. Не репортить формальные совпадения без реального влияния.
5. **Сохрани отчёт** в `docs/reviews/YY-MM-DD-audit-{scope}.md`.

## Чеклист: Код

- **Сложность:** длинные функции (>50 строк), глубокая вложенность (>4 уровня), god-файлы (>500 строк)
- **Дублирование:** повторяющаяся логика, копипаст валидации, одинаковые паттерны в разных модулях
- **Именование:** абстрактные имена (data, temp, value), несогласованные глаголы в одном модуле
- **Магические значения:** хардкод чисел/строк в бизнес-логике без именованных констант
- **Мёртвый код:** неиспользуемые импорты, функции, закомментированный код
- **Принципы:** нарушения DRY/KISS/YAGNI с контекстным анализом (не каждое дублирование — проблема)

## Чеклист: Архитектура

- **Слоистость:** нарушения границ слоёв, прямые вызовы в обход интерфейсов
- **Лишние абстракции:** абстрактные классы с одной реализацией, фабрики для <3 типов, обёртки без добавленной ценности
- **Связанность:** модули, которые знают слишком много друг о друге
- **Соответствие архитектурному документу:** реальный код vs описанная архитектура
- **Простота:** соответствие сложности решения сложности бизнес-задачи

## Чеклист: Тесты

- **Покрытие:** ключевые бизнес-пути имеют happy-path + error-path тесты
- **Качество:** тест читается как документация, одно поведение на тест
- **Изоляция:** I/O замокан, нет зависимости от порядка, времени, случайности
- **Граничные значения:** пустые коллекции, None, нуль

## Чеклист: Документация

- **Синхронность:** architecture.md отражает реальную систему
- **project_state:** актуальный, не раздутый
- **README:** точка входа с quickstart, ссылки рабочие

## Уровни findings

- **[КРИТИЧНО]** — баг, нарушение безопасности, сломанная архитектура. Требует исправления.
- **[ПРОБЛЕМА]** — ощутимо ухудшает качество, maintainability или читаемость. Рекомендуется исправить.
- **[УЛУЧШЕНИЕ]** — можно лучше, но работает. На усмотрение.

## Формат отчёта

```markdown
# Аудит: {scope}

**Дата:** YYYY-MM-DD
**Scope:** {полный | код | архитектура | тесты | документация}

## Общая оценка

Краткий вердикт и ключевые наблюдения (2-3 предложения).

## Сильные стороны

Что сделано хорошо (не пропускать — это важно для контекста).

## Findings

### [КРИТИЧНО] Название
- **Где:** файл(ы) и строки
- **Что:** описание проблемы
- **Почему важно:** влияние на проект
- **Рекомендация:** что сделать

### [ПРОБЛЕМА] Название
...

### [УЛУЧШЕНИЕ] Название
...

## Рекомендуемый порядок действий

Приоритизированный список из 3-5 конкретных шагов.
```

## Ограничения

- Не исправлять — только находить и описывать.
- Не создавать issues автоматически.
- Контекстный анализ: не репортить pattern match без подтверждения что это реальная проблема.
- Скилл универсальный — не привязан к конкретному проекту.
