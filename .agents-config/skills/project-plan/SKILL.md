# Нарезка задач

Раздели активную фазу проекта на независимые задачи, которые можно брать в работу без повторного изобретения scope.

## Когда использовать

- Есть PRD, architecture и phase-документ, и фазу нужно превратить в backlog.
- Нужно создать или обновить stories/issues текущей фазы.
- Нужно синхронизировать backlog с текущей фазой после изменения требований.

## Когда не использовать

- Фаза ещё не описана.
- Нужен план реализации одной конкретной задачи, а не backlog фазы.
- Нужен новый PRD, новая фаза или техдизайн.

## Процесс

1. Прочитай `docs/prd.md`, `docs/architecture.md` и phase-документы проекта.
2. Определи активную фазу:
   - используй явный статус, если он указан;
   - если фаза названа пользователем, считай её активной;
   - если фаз несколько и статус неочевиден, выбери наиболее релевантную текущему запросу и явно зафиксируй это допущение.
3. При необходимости дочитай существующие `docs/stories/`, `docs/issues/` и `project_state.md`, чтобы сохранить нумерацию, статусы и локальные конвенции.
4. Извлеки задачи только из требований активной фазы.
   Не превращай в backlog общие идеи из PRD, которые ещё не вошли в scope фазы.
5. Нарежь задачи по принципу: одна задача = один vertical slice ценности или одно ограниченное техническое изменение.
6. Для каждой задачи определи тип:
   - `story` для пользовательской или бизнес-ценности;
   - `issue` для багов, улучшений, tech debt, рефакторинга, инфраструктурных и прочих технических изменений без самостоятельной пользовательской ценности.
7. Продолжи существующую нумерацию.
   Если явной конвенции ещё нет, используй `US-XXX` для stories и `ISS-XXX` для issues.
8. Создай или обнови backlog-документы и синхронизируй `project_state.md` или локальный аналог состояния проекта.

Если в проекте уже есть устойчивая конвенция по путям, именованию или структуре backlog-документов, следуй ей вместо жёсткого навязывания дефолтной схемы.

## Правила нарезки

- Каждая задача должна быть независимо понятной и проверяемой.
- Предпочитай end-to-end slices, а не разбиение по слоям вроде "API отдельно, UI отдельно", если только это не осознанный технический issue.
- Не смешивай в одной задаче несколько независимых исходов.
- Если задача слишком большая, разделяй по пользовательскому сценарию, интеграционной границе, переходу состояния или error-path.
- Зависимости фиксируй явно, а не прячь внутри oversized-задач.
- Если требования фазы неоднозначны, выноси открытые вопросы и `TBD`, а не придумывай backlog из предположений.

## Story и Issue

### Story

Используй `story`, когда результат даёт понятную пользовательскую или бизнес-ценность и может быть сформулирован как роль, действие и цель.

### Issue

Используй `issue`, когда задача описывает:
- баг;
- улучшение существующего поведения;
- tech debt;
- рефакторинг;
- инфраструктурное или внутреннее изменение;
- технический enabler без самостоятельной пользовательской ценности.

Если есть сомнение, задай себе вопрос: "Если эту задачу сделать отдельно, получит ли пользователь или стейкхолдер законченную ценность?" Если да, это `story`. Если нет, это `issue`.

## Рекомендуемый результат

### Story

```markdown
# US-XXX: Название

**Phase:** {NN}-{slug}

## Description

Как <роль> я хочу <действие>, чтобы <цель>

## Acceptance Criteria

### Business
- [ ] Пользовательский или бизнес-результат

### Technical
- [ ] Тесты, обработка ошибок, логирование или другие обязательные техусловия

## Technical Notes

Короткие детали реализации, ограничения или зависимости.

## Related Files

- `path/to/file`
```

### Issue

```markdown
# ISS-XXX: Название

**Type:** bug | enhancement | tech_debt
**Priority:** high | medium | low
**Related US:** US-XXX или —

## Description

Описание проблемы или технической задачи.

## Technical Notes

Короткий технический контекст, ограничения или зависимости.
```

Структуру можно адаптировать под проект, если сохраняются те же смысловые поля.

## Обновление состояния проекта

- Если проект использует `project_state.md`, обнови его кратко и по факту.
- Отрази активную фазу, новые идентификаторы задач и текущее состояние backlog.
- Не превращай `project_state.md` в длинный журнал решений; для этого нужны отдельные документы.

## Финализация

- Покажи, какие backlog-документы созданы или обновлены.
- Коротко перечисли допущения, открытые вопросы и `TBD`, если они остались.
- Предложи следующий логичный шаг: начать с первой story, уточнить спорные задачи или сделать review backlog.
