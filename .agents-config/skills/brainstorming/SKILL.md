# Превращение идей в решения

Помоги превратить идею в проработанное решение через естественный диалог.

Начни с понимания контекста проекта, затем задавай вопросы по одному, чтобы уточнить идею. Когда поймёшь, что именно строим, представь решение и получи одобрение.

<HARD-GATE>
НЕ вызывай скиллы реализации, не пиши код, не создавай файлы проекта и не предпринимай никаких действий по реализации, пока не представишь решение и пользователь его не одобрит. Это касается ЛЮБОЙ задачи, независимо от кажущейся простоты.
</HARD-GATE>

## Антипаттерн: "Это слишком просто, чтобы проектировать"

Каждая задача проходит через этот процесс. Утилита на одну функцию, изменение конфига, небольшой рефакторинг — всё. "Простые" задачи — это именно то место, где непроверенные допущения приводят к максимальной переработке. Решение может быть коротким (несколько предложений для действительно простых задач), но ты ОБЯЗАН его представить и получить одобрение.

## Чеклист

Выполняй шаги строго по порядку:

1. **Изучи контекст проекта** — файлы, документация, недавние коммиты
2. **Задавай уточняющие вопросы** — по одному, пойми цель, ограничения, критерии успеха
3. **Предложи 2-3 подхода** — с trade-offs и своей рекомендацией
4. **Представь решение** — по секциям, масштабируя глубину под сложность; получи одобрение после каждой секции
5. **Зафиксируй решение** — сохрани в `docs/notes/YYYY-MM-DD-design-{slug}.md`
6. **Переход к следующему шагу** — предложи релевантный скилл

## Процесс

### Понимание идеи

- Сначала изучи текущее состояние проекта: `project_state.md`, релевантный код и документацию.
- Перед детальными вопросами оцени scope: если запрос описывает несколько независимых подсистем, сразу обозначь это. Не трать вопросы на детализацию проекта, который сначала нужно декомпозировать.
- Если задача слишком большая для одного решения, помоги пользователю разбить её на независимые части: какие это части, как они связаны, в каком порядке строить. Затем проведи brainstorming первой части через обычный flow.
- Для задач подходящего scope задавай вопросы по одному.
- Предпочитай вопросы с вариантами ответов, но открытые тоже допустимы.
- Только один вопрос за сообщение — если тема требует большего, разбей на несколько вопросов.
- Фокусируйся на понимании: цель, ограничения, критерии успеха.

### Исследование подходов

- Предложи 2-3 разных подхода с trade-offs.
- Представь варианты разговорно, со своей рекомендацией и обоснованием.
- Начинай с рекомендуемого варианта и объясни почему.

### Представление решения

- Когда считаешь, что понял, что строим, представь решение.
- Масштабируй каждую секцию под её сложность: несколько предложений, если всё очевидно, до 200-300 слов, если есть нюансы.
- После каждой секции спрашивай, всё ли выглядит правильно.
- Покрой: архитектуру, компоненты, поток данных, обработку ошибок, тестирование.
- Будь готов вернуться и уточнить, если что-то не сходится.

### Проектируй для изоляции и ясности

- Разбивай систему на небольшие юниты, каждый с одной чёткой задачей, общающиеся через явные интерфейсы, понятные и тестируемые независимо.
- Для каждого юнита должен быть ответ на: что он делает, как его использовать, от чего зависит.
- Можно ли понять юнит, не читая его внутренности? Можно ли изменить внутренности, не сломав потребителей? Если нет — границы нужно доработать.
- Меньшие, чётко ограниченные юниты также проще для LLM — рассуждения точнее, когда код помещается в контекст, и правки надёжнее, когда файлы сфокусированы.

### Работа в существующей кодовой базе

- Изучи текущую структуру перед предложением изменений. Следуй существующим паттернам.
- Если существующий код мешает работе (файл слишком разросся, нечёткие границы, запутанные ответственности), включи точечные улучшения в решение — как хороший разработчик улучшает код, в котором работает.
- Не предлагай несвязанный рефакторинг. Фокусируйся на том, что служит текущей цели.

## После одобрения решения

**Фиксация:**

- Запиши валидированное решение в `docs/notes/YYYY-MM-DD-design-{slug}.md`.
- Если в проекте есть другая устойчивая конвенция для design-документов, следуй ей.

**Переход к следующему шагу:**

Предложи пользователю релевантный следующий скилл в зависимости от результата:

- Решение описывает конкретную задачу → `implement-task`
- Нужна нарезка на backlog → `project-plan`
- Решение меняет архитектуру проекта → `tech-design`
- Решение описывает новую фазу → `new-phase`

Не переходи к реализации без явного подтверждения пользователя.

## Ключевые принципы

- **Один вопрос за раз** — не перегружай собеседника множеством вопросов.
- **Вопросы с вариантами** — на них проще отвечать, чем на открытые; используй, когда возможно.
- **YAGNI безжалостно** — убирай ненужные фичи из всех решений.
- **Исследуй альтернативы** — всегда предлагай 2-3 подхода перед выбором.
- **Инкрементальная валидация** — представляй решение по частям, получай одобрение перед продвижением.
- **Гибкость** — возвращайся и уточняй, если что-то не сходится.

## Ограничения

- Не начинай реализацию до одобрения решения.
- Не придумывай требования, которые пользователь не подтвердил.
- Для простых задач решение может быть коротким, но оно должно быть представлено и одобрено.
- Не предлагай абстракции "на будущее" — только то, что закрывает текущие требования.
