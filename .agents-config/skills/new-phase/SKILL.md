# Новая фаза

Сформируй phase-документ нового этапа roadmap без смешивания его с PRD всего
проекта, техдизайном или нарезкой задач.

## Когда использовать

- Пользователь хочет открыть следующую фазу roadmap.
- Нужно описать scope новой фазы, её границы и зависимости.
- Нужно зафиксировать новый этап в `docs/phases/` для уже существующего проекта.

## Когда не использовать

- Проект только стартует и ещё нет PRD.
- Нужен техдизайн новой системы, а не phase-документ.
- Нужна нарезка задач, а не определение scope фазы.
- Нужно лишь актуализировать документацию по уже реализованной работе.

## Процесс

1. Прочитай `docs/prd.md`, `docs/architecture.md`, существующие
   `docs/phases/*.md` и при необходимости `project_state.md`. Кратко зафиксируй,
   какой контекст учтён.
2. Задавай короткие уточняющие вопросы, по 1-3 за раз, и собери:
   - что войдёт в фазу;
   - что точно не войдёт;
   - какие требования и ограничения должны быть соблюдены;
   - меняются ли бизнес-требования.
3. Выяви зависимости, риски, edge cases и архитектурные последствия, которые
   пользователь мог упустить: совместимость с текущей функциональностью,
   миграции, rollout, операционные ограничения, влияние на текущую архитектуру.
4. Обнови roadmap в `docs/prd.md`, но не переписывай существующие бизнес-
   требования PRD, если пользователь явно не просил отдельное обновление PRD.
5. Создай `docs/phases/{NN}-{slug}.md`. Если номер фазы не задан, возьми
   следующий по порядку из существующих документов.

## Документ фазы

Документ фазы должен покрывать:

- что будет реализовано;
- подробные требования фазы;
- что не будет реализовано;
- опционально: изменения бизнес-требований;
- опционально: зависимости и риски;
- опционально: ожидаемые архитектурные изменения.

Если ответ неизвестен, фиксируй `TBD`, а не придумывай детали.
