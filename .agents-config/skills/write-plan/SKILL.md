# План реализации

Подготовь самодостаточный план, по которому агент без контекста проекта сможет
выполнить задачу после очистки сессии.

## Когда использовать

- После brainstorming, когда решение одобрено и нужно превратить его в план.
- Нужно спланировать доработку, рефакторинг или изменение, не привязанное к US/ISS.
- Пользователь явно просит подготовить план перед кодом.
- Задача нетривиальна и перед реализацией нужен явный scope и порядок шагов.

## Когда не использовать

- Задача привязана к US/ISS — тогда используй `implement-task`, который включает планирование.
- Нужен только brainstorming без фиксации плана.
- Задача тривиальна и план не нужен (быстрый фикс, правка конфига).

## Процесс

1. Прочитай контекст:
   - `project_state.md`;
   - design-документ из brainstorming, если он есть;
   - релевантный код и документацию.
2. Определи границы задачи:
   - что должно измениться;
   - что точно не входит в scope;
   - какие зависимости и внешние контракты затрагиваются.
3. Если есть неустранимая неоднозначность, задай 1-2 коротких вопроса.
4. Подготовь план в `docs/plans/YYYY-MM-DD-plan-{slug}.md`.
5. Сделай план самодостаточным: по нему другая LLM должна суметь выполнить задачу
   без повторного исследования всего проекта.
6. Остановись и дождись явного одобрения перед реализацией.

## Формат плана

```markdown
# {Название задачи}

> **Goal:** Одно предложение — что будет сделано и зачем.
> **Architecture:** 1-2 предложения — какие модули/слои затронуты
> и как это вписывается в систему.

## Acceptance Criteria

### Business
- [ ] Что пользователь сможет сделать или увидеть

### Technical
- [ ] Тесты, логирование, обработка ошибок

## File Structure

- Create: `exact/path/to/new_file.py`
- Modify: `exact/path/to/existing.py` — что именно меняется
- Test: `tests/exact/path/to/test_file.py`

## Контракт

Публичные интерфейсы, сигнатуры, входы/выходы,
поведение на граничных случаях.

## Детали реализации

Маппинг данных, константы, литералы, пороги, ключи
и другие значения, которые нельзя оставлять на усмотрение
при кодинге.

## Vertical Slices

### Slice 1: {Название}
- Какие файлы меняются
- Что именно делается
- Какие проверки запустить после

### Slice 2: {Название}
...
```

## Требования к качеству плана

### Самодостаточность

План — главный источник контекста для исполнителя. Он должен содержать всё,
что нужно для реализации, без перечитывания всей документации проекта:

- Header даёт быстрый вход: цель и архитектурный контекст за 3 строки.
- File Structure показывает полную карту изменений до начала работы.
- Детали реализации фиксируют всё, что легко потерять при очистке контекста.

### Что включать обязательно

- Acceptance criteria: бизнес и технические.
- File Structure: точные пути файлов, что создаётся, что меняется, где тесты.
- Контракт: публичные интерфейсы и поведение на граничных случаях.
- Vertical slices: порядок шагов с проверками после каждого.

### Чего не включать

- Готовый код — план описывает *что* делать, а не содержит реализацию.
- Edge cases, которые не требуются бизнесом.
- Абстракции "на будущее".
- Если есть reference-реализация, явно отделяй, что копируется без изменений,
  а что должно отличаться.

### Секции по необходимости

Включай только те секции, которые действительно нужны задаче.
Для простых задач достаточно Header, AC, File Structure и одного Vertical Slice.
Не раздувай план ради полноты шаблона.

## Ограничения

- Не смешивай план с реализацией: в этом скилле не меняй код.
- Не оставляй критичные детали как "доработать по месту".
- Не раздувай план абстракциями, если задача простая.
- Если есть несколько подходов, зафиксируй рекомендуемый и кратко укажи trade-off.
- Если часть решения спорная, пометь как `TBD`.

## Финализация

- Покажи путь к созданному плану.
- Коротко перечисли ключевые решения, допущения и открытые вопросы.
- Явно остановись и дождись одобрения пользователя перед реализацией.
