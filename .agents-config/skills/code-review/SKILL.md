# Код-ревью

Проверь изменения как независимый reviewer. Главный результат: конкретные findings,
которые можно исправить или осознанно отклонить.

## Когда использовать

- Пользователь просит проверить код, diff, commit, pull request или любые локальные изменения, даже если они не привязаны к `US-XXX` или `ISS-XXX`.
- Нужно проверить реализацию конкретной `US-XXX` или `ISS-XXX`.
- Пользователь просит сделать код-ревью изменений.
- Нужно сохранить review в `docs/reviews/` для дальнейшей доработки.

## Когда не использовать

- Пользователь просит придумать план реализации, а не ревью существующих изменений.
- Изменений нет и ревью нечего проверять.
- Пользователь просит аудит текущего состояния проекта, а не ревью конкретных изменений — это `audit`.

## Процесс

1. Прочитай `project_state.md`, чтобы понять текущую архитектуру, ограничения и грабли.
2. Если в запросе есть `US-XXX` или `ISS-XXX`, прочитай соответствующий story/issue файл.
3. Определи, какие изменения ревьюить: рабочее дерево (`git status && git diff`), конкретный diff, commit или файлы из запроса.
4. Прочитай изменённые файлы целиком, а не только отдельные фрагменты.
5. Пройди по чеклисту ниже.
6. Сформулируй findings по убыванию важности.
7. Сохрани review:
   - `docs/reviews/US-XXX-code-review.md`, если известен `US-XXX`;
   - `docs/reviews/ISS-XXX-code-review.md`, если известен `ISS-XXX`;
   - `docs/reviews/YY-MM-DD-code-review-{slug}.md`, если ID нет.

## Чеклист проверки

**Соответствие требованиям** (если был указан ID):
- [ ] Каждый пункт AC / описания issue отражён в коде
- [ ] Изменения строго в рамках задачи (нет самовольного рефакторинга)
- [ ] Затронуты только файлы, логически связанные с задачей

**Архитектура** — ВСЕГДА:
- [ ] Импорты и зависимости не нарушают слоистую архитектуру
- [ ] Код переиспользует существующие модули вместо дублирования
- [ ] Паттерн совпадает с аналогичными модулями проекта

**Документационные границы** (если затронуты docs) — ВСЕГДА:
- [ ] `docs/architecture.md` не превращён в операционный журнал
- [ ] `project_state.md` остаётся коротким (грабли-триггеры, без длинных разборов)
- [ ] Рабочие знания вынесены в `docs/notes/*.md` со ссылкой из релевантных документов

**Качество кода** — ВСЕГДА:
- [ ] Имена содержат термины предметной области (не data, temp, val)
- [ ] Конфигурационные параметры вынесены в константы/конфиги
- [ ] Нет магических чисел без пояснения

**Надёжность** — ВСЕГДА:
- [ ] Внешние вызовы (API, БД, файлы) обёрнуты в обработку ошибок
- [ ] Явно обработаны null, undefined, пустые коллекции
- [ ] Типы данных корректны

**Тестирование** — ВСЕГДА:
- [ ] Новая бизнес-логика покрыта: happy-path + edge/error-path
- [ ] Для багфикса есть regression-тест
- [ ] I/O замокан на границе — unit-тесты не ходят в реальные сервисы
- [ ] Тесты изолированы: один тест — одно поведение
- [ ] Тест читается как документация: по названию понятно что и при каком условии
- [ ] Граничные значения: пустые коллекции, None, нуль, максимумы
- [ ] Тесты детерминированы: нет зависимости от времени, случайности, порядка запуска

**Логирование** — ВСЕГДА:
- [ ] Вход в операцию залогирован с ключевыми параметрами (`info`)
- [ ] Бизнес-развилки (пропуск, fallback, фильтрация) — `warning`
- [ ] I/O-вызовы логируются с endpoint / размером ответа (`info`)
- [ ] Ошибки перед `raise` — `error` с контекстом для диагностики
- [ ] `error` — только реальные сбои; обработанные кейсы — `warning`/`info`
- [ ] Нет логирования секретов — только идентификаторы
- [ ] По логам восстанавливается цепочка: запуск → данные → фильтрация → результат
- [ ] Идентификатор операции/объекта в каждом сообщении для корреляции

**Безопасность** — ВСЕГДА:
- [ ] Нет секретов в коде (токены, пароли, ключи API)
- [ ] Входящие данные валидируются перед использованием

## Стандарт ревью

- Каждое замечание — факт, а не интерпретация ("не реализован AC X" — факт; "код мог бы быть лучше" — нет).
- Для каждого finding указывай файл и строку.
- **[БЛОКЕР]** — не выполнен AC, баг, нарушение архитектуры, безопасность, отсутствие тестов на новую логику. Требует исправления.
- **[ЗАМЕЧАНИЕ]** — качество, читаемость, edge cases, пробелы в логировании. Разработчик рассматривает и либо исправляет, либо обосновывает.
- Если findings нет — напиши это явно и укажи остаточные риски.

## Формат результата

```markdown
## Код-ревью: {ID или описание}

### Вердикт: {Готов к коммиту | Требует доработки}

### Замечания

1. **[БЛОКЕР]** **{file}**, строка {N}: Что не так.
   - Сейчас: `проблемный код или описание`
   - Нужно: `как должно быть`
   - Причина: почему это важно.

2. **[ЗАМЕЧАНИЕ]** **{file}**, строка {N}: Что не так.
   - Сейчас: `проблемный код или описание`
   - Нужно: `как должно быть`
   - Причина: почему это важно.
```
