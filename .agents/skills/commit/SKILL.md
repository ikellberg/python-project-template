# Commit

Создай коммит с осмысленным описанием на русском языке.

## Когда использовать

- Пользователь явно просит сделать commit.
- Нужно подготовить осмысленное сообщение по уже готовым изменениям.

## Когда не использовать

- Пользователь не просил commit.
- Scope изменений неясен из-за большого числа несвязанных файлов.

## Процесс

1. Посмотри `git status` и `git diff`, чтобы понять реальный scope.
2. Посмотри несколько последних commit messages, чтобы сохранить стиль проекта.
3. Если есть несвязанные изменения, stage только относящиеся к задаче файлы.
4. Сформируй короткое сообщение:
   - первая строка: `feat|fix|refactor|docs|chore: краткое описание`;
   - далее, если нужно, 1-3 коротких пункта с сутью изменений.
5. Если пользователь просил commit явно, выполни commit. Если scope спорный, сначала остановись и попроси подтвердить границы коммита.
6. После commit покажи, остались ли незакоммиченные изменения.

## Ограничения

- Если коммит завершает задачу (US/ISS) — убедись, что `PROJECT_STATE.md` обновлён (спринт, бэклог, грабли).
- Не используй `git add .`, если в рабочем дереве есть лишние файлы.
- Не делай push, если пользователь не просил.
- Не смешивай в одном commit завершённую задачу и unrelated changes.
