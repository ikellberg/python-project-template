# RULES v2

Базовый rulebook для Python-проектов, где используются `PRD`, phase-документы,
архитектурный документ и короткий `project_state`.

Если в конкретном проекте пути или названия документов отличаются, сохраняй
роли документов и принципы из этого файла, а не буквальные пути.

**Главный принцип:** решения должны быть простыми, проверяемыми и понятными
человеку и LLM-агенту.

---

# 1. Инженерные правила

## 1.1 Код и типы

- Публичные функции и контракты имеют аннотации типов.
- Используется современный синтаксис типов: `list[str]`, `str | None`.
- Для внешних контрактов, конфигурации и валидации предпочитай `Pydantic` или эквивалентную схему с явной валидацией.
- `dataclass` допустим для простых внутренних структур без сложной валидации.
- Для файлов использовать `pathlib.Path` и `encoding="utf-8"`.
- Числа хранятся как числа, отсутствие значения выражается через `None`, а не через магические значения.

## 1.2 Читаемость

- Одна функция или метод отвечают за одну явную задачу.
- Предпочитай guard clauses вместо глубокой вложенности.
- Если вложенность уходит слишком глубоко, разбивай код на более мелкие функции.
- Названия должны выражать предметную роль сущности, а не быть абстракциями вроде `data`, `temp`, `value`.

## 1.3 Архитектура

- Не создавать абстракции "на будущее", пока нет реальной потребности.
- I/O отделяется от бизнес-логики.
- Для сервисов по умолчанию использовать слои `entrypoints -> domain -> infra`, если проект не требует другой структуры.
- Контракты на границах системы должны быть явными и проверяемыми.
- Устойчивые архитектурные решения фиксируются в архитектурном документе, а не остаются только в коде.
- По умолчанию выбирать самое простое решение, которое закрывает текущие явные требования.
- Оптимизировать архитектуру под основной сценарий, а не под редкие operational edge cases (restart, race, manual overlap).
- Если предлагается более сложное решение, сначала кратко объяснить, почему простой вариант недостаточен.

## 1.4 Работа с LLM

Принцип: **Dumb Pipe -> Smart Reasoner -> Deterministic Output**

- LLM принимает решения и генерирует текст, но не подменяет валидацию и парсинг на границах системы.
- Входы и выходы LLM должны проходить через явный контракт: `Pydantic`, JSON Schema или другой проверяемый формат.
- Формат ответа должен быть задан заранее; поля "из головы" недопустимы.
- Нельзя полагаться на regex-парсинг свободного текста, если можно использовать структурированный ответ.
- Невалидный ответ LLM должен приводить к явной ошибке или повторной попытке, а не к угадыванию смысла.

## 1.5 Ошибки и fallback

- Валидируй входы на границах системы.
- Не подавляй ошибки молча.
- При повторном выбросе сохраняй контекст через `raise ... from e`, если это применимо.
- Fallback допустим только как осознанное поведение с явной логикой и логированием.

## 1.6 Логирование

- Использовать `logging` или эквивалентную системную библиотеку.
- Никогда не логировать секреты: токены, ключи, пароли, приватные данные.
- В логах оставлять только идентификаторы и безопасный диагностический контекст.

Минимум логирования:

- Вход в важную операцию с ключевыми параметрами.
- Бизнес-развилки: пропуск, fallback, фильтрация, деградация.
- Внешние I/O-вызовы: endpoint, размер ответа, количество сущностей.
- Ошибки перед `raise` с достаточным контекстом для диагностики.
- Интеграции и orchestration-потоки логируются на границах так, чтобы было видно запуск, ключевые развилки и итог.

По логам должна восстанавливаться цепочка:
запуск -> получение данных -> фильтрация/развилка -> результат или ошибка.

Если важный runtime-flow нельзя восстановить по логам, это дефект качества, а не необязательная доработка.

## 1.7 Тестирование

- Тесты пишутся вместе с кодом, а не после.
- Любая новая функциональность или изменение наблюдаемого поведения требует тестов.
- Новая бизнес-логика требует минимум happy-path и edge/error-path.
- Для багфикса обязателен regression-тест.
- Unit-тесты мокают I/O на границе.
- Smoke и ручные проверки не заменяют unit-тесты.
- Если тесты не удалось запустить, это фиксируется явно.
- Задача не считается завершённой с красными тестами без явного согласованного исключения.

## 1.8 Перед коммитом

- Линтеры и тесты проходят, либо причина невозможности запуска зафиксирована явно.
- Нет закомментированного мёртвого кода и временных костылей без объяснения.
- `TODO` допустим только вместе с причиной или ограничением.
- Если менялись переменные окружения или их контракт, перед завершением задачи нужно проверить, что с текущим кодом синхронизированы:
  - `.env.example`;
  - реальное целевое окружение, где это применимо: локальный `.env`, VPS `.env`, CI/CD secrets или их аналог.
- Если агент не может проверить или обновить целевое окружение сам, он обязан явно вынести это в ручные шаги и перечислить, что именно осталось несинхронизированным.

# 2. Процесс работы

## 2.1 Стоп-кран

- Если несколько попыток подряд не дают нового сигнала и не меняют понимание проблемы, нужно остановиться и пересобрать подход.
- Практическая эвристика по умолчанию: после 3 неудачных попыток считать, что агент, скорее всего, застрял.
- После остановки кратко фиксируются проблема, проверенные подходы, что именно не сработало, и следующая разумная альтернатива.

## 2.2 Планирование

- План перед реализацией нужен, если задача:
  - меняет архитектурные границы, ключевые контракты или модель данных;
  - затрагивает несколько модулей или слоёв;
  - включает внешнюю интеграцию, миграцию, деплой-контур или другое изменение с повышенным риском;
  - не имеет очевидного пути реализации без предварительной декомпозиции.
- Для мелких правок, точечных doc-обновлений, isolated bugfixes и простых локальных изменений формальный план не обязателен.

Базовый цикл для существенных задач:

1. Исполнитель готовит план.
2. План проходит независимое ревью.
3. Исполнитель дорабатывает план по замечаниям.
4. Только потом начинается реализация.

Если проект использует структуру `docs/plans/`, планы хранятся там.
Имена по умолчанию:

- `{ID}-{slug}.md`;
- если ID нет, `YY-MM-DD-plan-{slug}.md`.

## 2.3 Разработка

Базовый цикл:

1. Исполнитель делает реализацию.
2. Исполнитель добавляет или обновляет тесты.
3. Код проходит независимое ревью.
4. Исполнитель отвечает на замечания и вносит правки.
5. Исполнитель выполняет финальную проверку и убеждается, что задача завершена, тесты проходят, а принятые замечания закрыты.

## 2.4 Ревью

- Ревью должно быть независимым от исполнителя изменения.
- Независимое ревью может выполняться другим агентом, человеком или отдельным review-pass, но не должно сводиться к формату "исполнитель сам себя одобрил".
- Каждое замечание оценивается критически, а не исполняется автоматически.
- Некорректные или избыточные замечания можно отклонять с обоснованием.
- Спорные изменения лучше обсуждать до внесения правок.
- Финальная ответственность за техническое решение остаётся у исполнителя.

Если проект использует `docs/reviews/`, ревью хранится там.
Имена по умолчанию:

- `US-XXX-plan-review.md`;
- `US-XXX-code-review.md`;
- `ISS-XXX-code-review.md`;
- если ID нет, `YY-MM-DD-review-{slug}.md`.

# 3. Система документации

## 3.1 Источники правды

- `PRD` — продуктовый источник правды: проблема, решение, бизнес-требования, roadmap, non-goals.
- `Phase`-документ — scope конкретной фазы: что будет реализовано и что сознательно не входит в фазу.
- `Architecture`-документ — главный инженерный документ проекта. Первая версия фиксируется на старте проекта, далее документ поддерживается в актуальном `as-is` состоянии.
- `project_state` — короткий стартовый контекст проекта, который помогает LLM быстро войти в новую сессию без повторного чтения всей документации. Это briefing-документ, а не отдельный источник правды.
- `Notes` — исследования, проверки, API-наблюдения, миграционные детали, справочные детали и рабочая база знаний проекта.

## 3.2 Backlog-артефакты

- `Stories` описывают пользовательскую или бизнес-ценность.
- `Issues` описывают баги, tech debt, инфраструктурные и внутренние технические задачи.
- По умолчанию backlog-артефакты относятся к активной фазе.
- Если есть межфазовый техдолг или отдельный глобальный backlog, это должно быть помечено явно, а не подразумеваться.

## 3.3 Роли документов по умолчанию

- `docs/prd.md` — продукт целиком.
- `docs/phases/{NN}-{slug}.md` — конкретная фаза.
- `docs/architecture.md` — архитектура и ключевые контракты текущей системы.
- `docs/stories/` — stories текущего активного backlog.
- `docs/issues/` — issues текущего активного backlog или явно помеченного техдолга.
- `docs/plans/` — планы реализации.
- `docs/reviews/` — ревью.
- `docs/notes/` — длинные рабочие знания.
- `project_state.md` — короткий стартовый briefing проекта для новой сессии.
- `README.md` — точка входа в репозиторий: краткий контекст, ссылки на основные документы и quickstart/run/deploy; не источник правды по требованиям, архитектуре или статусу.

Если проект использует другие пути, сохраняй эти роли, а не эти буквальные имена.

## 3.4 Правила маршрутизации

- Исследования, проверки, справочные детали и рабочая база знаний проекта -> `notes`.
- Устойчивое техническое решение -> `architecture`.
- Продуктовая граница или roadmap -> `PRD`.
- Scope конкретной фазы -> `phase`-документ.
- Короткий стартовый контекст сессии -> `project_state`.
- План реализации задачи -> `plans`.
- Findings и замечания -> `reviews`.

## 3.5 Требования к project_state

`project_state.md` или его локальный аналог — это стартовый briefing для новой сессии. Его цель: дать рабочий контекст для LLM без перечитывания всей документации.

Документ должен содержать только высокий сигнал:

- краткий раздел `Бизнес-контекст`: какую проблему решает система и в чём её основная ценность;
- краткий раздел `Архитектура`: ключевые модули, поток данных и главные интеграции;
- краткий раздел `Текущий бэклог задач`: ближайшие актуальные задачи или явное указание, что активных задач нет;
- раздел `Грабли`: неочевидные решения и ограничения, которые легко забыть и дорого переоткрывать.

Опционально:

- короткий блок `Недавние изменения`, если без него трудно понять текущее состояние проекта.

Правила:

- обновлять после принятой завершённой задачи, а не в промежуточных циклах разработки;
- не дублировать `PRD`, `architecture`, `stories`, `issues` и структуру файлов целиком; в `project_state` допустим только короткий список актуальных задач;
- не превращать документ в changelog, архив `done` или сокращённую копию `architecture.md`;
- держать структуру простой: бизнес-контекст -> архитектура -> текущий бэклог задач -> грабли;
- пустые обязательные секции не нужны: если информации нет, достаточно коротко указать это или опустить раздел;
- оставлять только то, что реально помогает быстро войти в проект.
